hReader Coding Guidelines

This will serve as the style guide for all source code checked into hReader. This expands on Apple's guidelines.

General
  • Group system #import statements together.
  • Group all other #import statements by like prefix.
  • Prefer @class and @protocol to #import when referring to external classes in a public @interface.
  • Prefer @property for public attributes to declaring instance variables with accessors and mutators.
  • Prefix class, protocol, and file names with a 2- or 3-letter identifier.
  • Order @property modifiers: atomicity, memory management, writability.
  • Separate methods by function using #pragma mark - <Section Name>
  • Separate sections into subsections using #pragma mark <Subsection Name>
  • Use @[ arrayObject ], @{ @"key" : value }, @(YES or NO), and @5.0 literals.
  • Use @interface ClassName () to declare private properties.
  • Use lowerCamelCase for method names.
  • Use NSAssert and NSParameterAssert in methods that require the presence of certain arguments.
  • Declare internal constants using static and const.
  • Declare external constants using extern and const.
Brackets

Always place opening brackets on the same line as the opening statement. Always use brackets on blocks of code. - (void)foo {

} if (foo) {

}

Always place closing brackets on their own line. if (foo) {

}
else {

}

It is acceptable to place the body of a block of code on one line in certain cases. if (foo) { return; }

Indent style
K&R C indent style is preferred. When adhering to K&R, each function has its opening brace at the next line on the same indentation level as its header, the statements within the braces are indented, and the closing brace at the end is on the same indentation level as the header of the function at a line of its own. The blocks inside a function, however, have their opening braces at the same line as their respective control statements; closing braces remain in a line of their own, unless followed by an else or while keyword.
Identifier Naming

Stack variables are named using lower-camel-case, and should form a noun. int someVariable = 0;

The asterisk used for pointer declarations should be prefixed to the variable identifier. NSString *someString;

C functions are prefixed with an identifier and are named in upper-camel-case. void ABCDoSomething(BOOL argument);

Commenting

While comments are highly encouraged for documenting details and reasonings that are not obvious from the code itself, comments should never be considered a replacement for clear code. Before adding a comment, ask yourself if the code itself could be clarified, removing the need for the comment altogether.

Don't do:

int nEntsW; // Number of entities in the world

Do:

int numberOfEntitiesInTheWorld;

Whitespace
  • 4 spaces for indentation - no tabs.
  • End files with a newline.
Documentation
  • All public method declarations should be documented.
  • Comments should be hard-wrapped at 80 characters.
Declarations
  • Prefer exposing an immutable type for a property if it being mutable is an implementation detail. This is a valid reason to declare an ivar for a property.
  • Always declare memory-management semantics even on readonly properties.
  • Declare properties readonly if they are only set once in -init.
  • Don't use @synthesize unless the compiler requires it. Note that optional properties in protocols must be explicitly synthesized in order to exist.
  • Instance variables should be prefixed with an underscore (just like when implicitly synthesized).
Expressions
  • Don't access a property-declared ivar unless you're in -init, -dealloc, or an overridden accessor.
  • Only use dot-syntax on methods that are declared as properties.
  • Comparisons should be explicit for everything except BOOLs.
  • Prefer positive comparisons to negative.
  • Long form ternary operators should be wrapped in parentheses and only used for assignment and arguments.
  • Short form, nil coalescing ternary operators should avoid parentheses.
  • There shouldn't be a space between a cast and the variable being cast.
Control Structures
  • Always surround if bodies with curly braces. Single-line if bodies without an else should be on the same line as the if.
  • All curly braces should begin on the same line as their associated statement. They should end on a new line.
  • Put a single space after keywords and before their parentheses.
  • Return and break early.
  • No spaces between parentheses and their contents.
Blocks
  • Blocks should have a space between their return type and name.
  • Block definitions should omit their return type when possible.
  • Block definitions should omit their arguments if they are void.
Literals
  • The contents of array and dictionary literals should have a space on both sides.
  • Longer or more complex literals should be split over multiple lines.
Categories
  • Category methods should always be prefixed.
  • If you need to expose private methods for subclasses or unit testing, create a class extension named Class+Private.
